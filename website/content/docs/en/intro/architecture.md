# Architecture

UniRhy is organized as a monorepo with three core sub-projects plus external plugin packages. This overview helps with deployment decisions and troubleshooting.

## Components

### server (Spring Boot backend)

- Stack: Spring Boot 4 · Kotlin · Gradle · JVM 25 · Jimmer ORM · PostgreSQL.
- Serves REST APIs, the playback-sync WebSocket channel, and signed media file access.
- Persists all music metadata, accounts, playback preferences, current queues and resume state.
- Schema migrations are managed by Flyway.

### web (Vue 3 frontend / Tauri 2 desktop)

- Stack: Vue 3 · TypeScript · Vite · Pinia · Tailwind CSS · Tauri 2.
- Runs as a browser client out of the box; bundled with Tauri it ships as macOS / Windows / Android / iOS apps.
- The API client (`/web/src/__generated`) is generated by Jimmer at build time.

### website (this site)

- Stack: Vue 3 · Vite · SSR prerendering.
- The site you are reading. Deployed to Cloudflare Workers.

### plugins (external)

- Separate repositories (default set under `unirhy-plugins`), loaded by the server via a defined interface.
- Used for metadata enrichment, artist name normalization, third-party audio processing and so on.

## Data flow

```
┌────────────┐        REST + WS         ┌─────────────┐
│   Client   │ ◀─────────────────────▶ │   Server    │
│ Web / Tauri│                          │ Spring Boot │
└────────────┘                          └──────┬──────┘
                                               │
                                  ┌────────────┼────────────┐
                                  ▼            ▼            ▼
                            ┌──────────┐  ┌─────────┐  ┌─────────┐
                            │PostgreSQL│  │ Storage │  │ Plugins │
                            │  (data)  │  │ Local / │  │  JARs   │
                            └──────────┘  │   OSS   │  └─────────┘
                                          └─────────┘
```

- **REST**: every CRUD / listing / configuration call goes over HTTP, with Sa-Token Cookie-based session.
- **WebSocket**: a dedicated `/ws/playback-sync` channel carries play / pause / seek / queue-change broadcasts.
- **Storage**: audio and artwork files live on configurable storage nodes — a local directory or S3-compatible OSS.
- **Plugins**: dynamically loaded JARs that implement the agreed interfaces for metadata, normalization, etc.

## Deployment shape

The minimal deployment needs only two processes:

- one Spring Boot process (server)
- one PostgreSQL instance

The frontend build (`web/dist`) is served directly by the server as static resources — no separate nginx is required. Front a reverse proxy if you need TLS or multi-port routing. See [Docker installation](/en/docs/install/docker) for details.
